# 🧠 Как это всё связано

Мы строим систему так:

- Clean Architecture → структура
- DDD-lite → моделирование бизнеса
- Hexagonal → изоляция инфраструктуры
- Event-driven → слабая связность
- Outbox → надёжность
- CQRS → масштабирование

Если упростить до уровня middle:

- Делай слои
- Не мешай ORM с бизнес-логикой
- Используй UoW
- Делай domain events
- Делай outbox
- Делай locking

# 3️⃣ Поток бронирования (highload-safe)

### Booking flow:

1. Получаем distributed lock (Redis)
2. Начинаем транзакцию
3. SELECT ... FOR UPDATE
4. Проверяем available_tickets
5. Создаём booking
6. Уменьшаем available_tickets
7. Пишем event в outbox
8. Commit
9. Фоновый воркер публикует событие
10. Отправка email / payment processing
